# 增强调用链影响范围分析功能 - 任务清单总结

我已经根据刚才制定的设计计划，以及您选择的 **方案 B**，完成了全部核心代码的升级替代。本次改动旨在让报告的内容更加精准、对应关系更清楚，并正式接入了“影响点自动解析大模型”。

## 核心实现总结

### 1. 完善顶层溯源能力
在 `models.kt` 中创建了全新的 `EntryPointType.OTHER` （⚙️其他顶级方法）。
重构了 `CallTree` 的属性抓取路线，现在，当调用树探索到顶层，且发现自身没有匹配到常规入口点注解（例如不在被标记为 Spring 或者 MQ）时，将自动把自己标记为 `OTHER`，确保在最终的**受影响入口点汇总表**里无所遗漏！

### 2. 构建链路表对齐功能
重构了 `ReportGenerator.kt` 下的表格生成范式，从原来使用暴力的 `flattenTree`（只展示扁平节点数组） 改为了通过新注入的 `collectChainTrees` 获取具备结构语义的全链路。最终在表格输出了**完整的链路编号、链路深度和起始请求节点**，完美实现了和打印日志一一对应的效果。

### 3. LLM 元信息自动注入
现已通过 `CodeSenseSettings.getInstance().getDefaultProvider().displayName` 实时提取用户选择的大模型（如 *Minimax* 或者 *GLM*），并将该 `displayName` 放进报告的 `metadata` 数据缓存中，成功渲染进界面底端的**分析元信息**表框。在也没有“Model: 未使用”的尴尬情况了。

### 4. 落地双流并发解析（方案B生效）
在 `AnalyzeImpactAction.kt` 重塑了整个大模型请求发送架构。现在当用户要求进行评估时，系统会启动两个轻量级协程并发请求大模型：
- **流式监听线程**：依然处理着原本的底部“综合功能评估”。
- **单发提取线程**：将整个所有 `EntryPointInfo` 提供给专门针对简略评论打造的新 Prompt；大模型会返回带编号的前置字段（类似于 `ENTRY_1: 某某方法会报错`），我们将该响应过滤解析后，逐一塞回到原本在原位标明 `待 AI 生成` 的位置。两个线程共用 `synchronized` 面板渲染锁，实现了“双层异步动态替换 Markdown”。
